# Phase 260428-l47 Plan 01: Pin deepseek-v4-flash Preset to DeepSeek Summary

**One-liner:** Changed `deepseek-v4-flash` provider field from `""` to `"deepseek"` in both `MODEL_PRESETS` (backend) and `DEFAULT_MODEL_PRESETS` (backend_generalist) so requests route through DeepSeek's first-party OpenRouter endpoint and avoid AkashML 502s, while leaving `allow_fallbacks=True` plumbing untouched.

## What Changed

Two single-line substitutions, identical in shape:

**`backend/agent/definition.py`** (line 17, inside `MODEL_PRESETS`):
```diff
-    "deepseek-v4-flash": ("deepseek/deepseek-v4-flash", ""),
+    "deepseek-v4-flash": ("deepseek/deepseek-v4-flash", "deepseek"),
```

**`backend_generalist/agent.py`** (line 116, inside `DEFAULT_MODEL_PRESETS`):
```diff
-    "deepseek-v4-flash": ("deepseek/deepseek-v4-flash", ""),
+    "deepseek-v4-flash": ("deepseek/deepseek-v4-flash", "deepseek"),
```

The string `"deepseek"` flows through `build_model_settings` / `_build_model_settings` (split on `,`) into `OpenRouterModelSettings.openrouter_provider.order = ["deepseek"]` with `allow_fallbacks: True`. No code changes were needed in the settings builders — that plumbing already existed.

## Why

After commit `8f0a852` (260428-kuu) added the `deepseek-v4-flash` preset with an empty provider field, OpenRouter routed requests through AkashML, which returned `502 Backend request failed`. DeepSeek's first-party endpoint reports 100% uptime per OpenRouter's endpoints API, so pinning the order to `["deepseek"]` selects the healthy provider while still allowing fallback to other providers (`allow_fallbacks: True`) if DeepSeek itself goes down.
